# Pre-Training Correctness Validation Blueprint

## Purpose

- Define the executable validation plan that must pass before formal neural-network optimization starts.
- Convert the first runnable loop from `TASK-0006` into a checked, source-traceable, audio-verifiable pipeline.
- Make failures actionable: every gate must either pass, trigger a project-side fix, or block training with a precise cause.

## Validation Harness Contract

- Preferred module:
  - `bsm.phase02.correctness_validation`
- Preferred CLI:

```bash
conda run -n bsm_harness_py311 python -m bsm.phase02.correctness_validation audit --indent 2
```

- Optional CLI parameters:
  - `--artifact-dir`
  - `--skip-audio`
  - `--reference-rotation-yaw-deg`
  - `--indent`
- Default artifact root:
  - `06_Assets/Generated_Artifacts/TASK-0007/<timestamp>/`
- The audit command must return non-zero if a required validation gate fails.

## Gate 2. BSM-MagLS Coefficient Parity

- Reference:
  - `compute_emagls_filters/compute_emagls2_for_rotations.py`
  - `compute_emagls_filters/emagls_32kHz_dft_aligned_ypr_0_0_0.npy`
  - `compute_emagls_filters/emagls_32kHz_dft_aligned_ypr_90_0_0.npy`
  - `bsm.phase02.front_end_bundle`
- Canonical axis policy:
  - project coefficient shape is `[frequency, microphone, ear]`
  - saved reference shape `(5, 2, 513)` is treated as `[microphone, ear, frequency]` unless the source code proves otherwise
  - exported summaries must record the source and canonical shapes
- Required metrics:
  - `max_abs`
  - `mean_abs`
  - `rmse`
  - `nmse`
  - per-ear metrics
  - per-frequency worst bins
- Difference localization order:
  - HRTF SH loading, resampling, and delay alignment
  - right-ear SH sign convention
  - array SH loading and mic-delay alignment
  - optimization grid direction order
  - LS covariance and diagonal loading
  - MagLS phase recursion above `2000 Hz`
  - diffuse-field equalization
  - saved reference rotation and yaw convention
- Acceptance:
  - direct parity passes within documented tolerance, or
  - project code is corrected and parity reruns pass, or
  - a remaining mismatch is recorded as a training blocker with a narrowed cause.

## Gate 2A. Orientation Coefficient Bank Contract

- Reference:
  - Berebi et al. 2025 BSM-iMagLS head-tracked evaluation
  - Madmoni et al. 2025 head-rotation BSM formulation
  - `compute_emagls_filters/emagls_32kHz_dft_aligned_ypr_*`
- Required semantic model:
  - every supported listener head orientation has its own coefficient set
  - for BSM-iMagLS, each orientation-specific coefficient set is generated by a DNN-based solver for that orientation's rotated/counter-rotated HRTF target
  - runtime playback or inference selects the precomputed coefficients from the head-tracker yaw/pitch/roll state
- Required bank axes and metadata:
  - public coefficient object remains `[frequency, microphone, ear]`
  - coefficient bank adds an orientation axis or keyed lookup over yaw/pitch/roll
  - each entry records yaw, pitch, roll, coefficient authority, source/reference path if applicable, solver settings, and validation metrics
- Non-goals:
  - no static yaw `0` coefficient reuse for all rotations
  - no claim that one trained DNN generalizes over arbitrary orientation unless explicitly validated in a later experiment
  - no online DNN optimization during playback
- Acceptance:
  - yaw `0` and yaw `90` coefficient entries can be selected explicitly
  - selected yaw `0` preserves the repaired saved-reference parity contract
  - selected yaw `90` either passes against the saved/generated yaw `90` authority or remains a documented rotation blocker
- Current implementation status:
  - saved-eMagLS yaw `0` and yaw `90` entries are implemented through `front_end_bundle.orientation_coefficients`
  - selected yaw `0` and selected yaw `90` both pass saved aligned-ypr parity
  - DNN-solver-optimized BSM-iMagLS entries per orientation remain a later coefficient-generation layer
  - TASK-0008 is the active task that must wire selected orientation entries through the neural training-path smoke before TASK-0009 large optimization

## Gate 3. Renderer Parity

- Reference:
  - `bsm.phase02.baseline_renderer.render_coefficients`
  - `bsm.phase02.residual_solver.render_response_torch`
  - `Array2Binaural/ild_itd_analysis/evaluate_ilds_itds.py`
- Check:
  - render `c_ls`
  - render project `c_magls`
  - render canonicalized Array2Binaural eMagLS filters when shape-compatible
  - compare numpy and torch renderer outputs on the same coefficients
  - compare rendered outputs to target `h`
- Required metrics:
  - response shape
  - finite flags
  - renderer numpy/torch `max_abs` and NMSE
  - target NMSE
  - normalized magnitude error
  - per-direction worst cases
- Acceptance:
  - numpy and torch renderer parity is within numerical precision
  - reference filters and project filters render through the same documented axis semantics
  - any response mismatch is traced to coefficient differences, not renderer implementation errors.

## Gate 4. Cue-Bank Validation

- Reference:
  - `bsm.phase02.cue_bank`
  - `05_Experiments/EXP-0001_Auditory_ILD_Python/code/auditory_ild.py`
  - `Array2Binaural/ild_itd_analysis/evaluate_ilds_itds.py`
  - ITD-loss paper Eq. 15 and Eq. 16
- Check:
  - ERB-band ILD metrics for target vs project `BSM-MagLS`
  - ERB-band ILD metrics for target vs Array2Binaural eMagLS reference, when renderable
  - GCC-PHAT ITD proxy metrics for the same responses
  - differentiable ILD and ITD loss helpers produce finite backward pass in focused tests
- Acceptance:
  - exported cue arrays and scalar metrics are finite
  - ILD path remains the ERB-band evaluation authority
  - differentiable broadband ILD proxy remains documented as an optimization helper, not the final evaluation definition.

## Gate 5. Solver And Export Readiness

- Reference:
  - `bsm.phase02.residual_solver`
  - `TASK-0006` exported summary shape
- Check:
  - solver input pack still uses the canonical `c_ls`, `c_magls`, and `c_magls - c_ls` semantics
  - residual output matches `c_magls`
  - loss loop forward and backward are finite
  - short-run smoke still exports TASK-0006-style fields
- Acceptance:
  - short-run smoke exits successfully
  - exported fields include baseline name, ILD error, ITD proxy error, normalized magnitude error, NMSE, selected iteration, and loss trace
  - no downstream code path consumes stale or incorrectly oriented coefficients.

## Artifact Schema

- `validation_summary.json`:
  - top-level `ok`
  - `schema_version`
  - `producer_task_id = TASK-0007`
  - `artifact_refs`
  - gate status list
  - blocker list
- `parity_metrics.json`:
  - reference path
  - source shape
  - canonical shape
  - project shape
  - metrics by ear and total
- `render_metrics.json`:
  - renderer parity metrics
  - target comparison metrics
  - worst direction records
- `cue_metrics.json`:
  - ILD and ITD metrics by rendered candidate
  - cue shapes
  - finite flags
- `audio_manifest.json`:
  - file path
  - label
  - case
  - sample rate
  - duration
  - peak amplitude
  - source response used
- `listening_notes.md`:
  - one row per WAV or paired comparison
  - channel swap, coloration, image direction, ITD stability, level balance, clipping, obvious artifacts

## Close Criteria

- All required gates are executed.
- All failures are fixed, or the task is moved to blocked with a precise blocker.
- Full discovered tests pass after fixes.
- Generated artifacts are registered.
- Human-listening notes exist and are linked.
- Training remains disallowed until `TASK-0007` closes as completed.
